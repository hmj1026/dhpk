## How it works

- **Phase A**: cumulative reference count ≥ 3 across sessions → entry listed below
- **Phase B confidence scoring** (start `0.5`, clamped to `[0.0, 1.0]`):
  - `+0.1` per clean session (no trap re-occurrence signal near the reference)
  - `-0.2` per trap re-occurrence (`踩` / `又` / `再` near a reference ±20 lines)
- **Time-span gate** (schema v2): promotion to `rule` / `skill` also requires
  - `last_seen − first_seen ≥ 24h` (env `CLAUDE_HOOK_MIN_SPAN_HOURS` to override)
  - `seen_dates` set size ≥ 3 distinct dates (env `CLAUDE_HOOK_MIN_DISTINCT_DATES`)
  - blocks the "3 citations in one day → graduate" false positive
- **Existence gate**: the source memory file must exist at
  `~/.claude/projects/<slug>/memory/<entry>.md`. Missing → marked `orphan`,
  confidence `-0.3` per round; on reaching 0 the `count -1`, and a zeroed count
  is tombstoned (removed). Low-confidence entries can tombstone on the first
  round (no "minimum N rounds" guarantee).
- **suggested_target**:
  - `orphan` → Orphan Entries section (not graduated)
  - `count ≥ 3` and `confidence ≤ 0.2` → `decay`
  - `count ≥ 3` and `confidence ≥ 0.7` and time-span gate passed → `rule`
  - `count ≥ 3` and `0.2 < confidence < 0.7` and time-span gate passed → `skill`
- **Test mode**: `CLAUDE_HOOK_TEST_MODE=1` redirects state into
  `$CLAUDE_HOOK_TEST_OUTDIR`, so smoke tests never write into the real
  `.claude/artifacts/` tree.

## Manual workflow

1. Run `dhpk:rules-distill` (it reads this file as Phase 1 supplemental input)
2. For `rule` candidates: decide whether to pull into `.claude/rules/` and
   remove from memory
3. For `decay` candidates: confirm with the author, then remove from memory
   (or rewrite for clarity)
4. For `orphan` candidates: residual records; once the source file is confirmed
   gone, let them tombstone automatically
